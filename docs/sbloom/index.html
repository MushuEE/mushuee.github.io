Probabilistic Search: Fast Negative Gates

Semantic Bloom Filter: A Journey from Bloom to Bust

Traditional Bloom Filters work for exact matches. Semantic Bloom Filters extend this to high-dimensional vector spaces, allowing for fast, probabilistic negative lookups to protect expensive databases.

@gmccloskey Workspace
May 2026 · Evolution of a Semantic Bloom Filter
Project Summary Lua vs MIH vs LSH

What we did: We explored three approaches to avoid heavy database queries for misses:
1. Brute Force Lua: Slow and blocking.
2. Multi-Index Hashing: Fast but too strict for loose similarity.
3. In-Memory LSH: Pure in-app gate, but introduces false positives when reducing false negatives.

Key Takeaway

There is no free lunch. To capture loose semantic similarity without full scans, you must accept either higher memory usage (HNSW) or false positives/negatives (LSH).

§1 · The Problem

The Cost of Semantic Search

Traditional caching mechanisms fail for semantic queries. We need a fast negative gate.

■ CHALLENGEVector DB overhead

Checking a Vector DB for every query introduces latency and infrastructure cost, even when the query is a miss. We need a Semantic Bloom Filter to say "definitely no match" in microseconds.

■ KEY INSIGHT The Cost Asymmetry Principle

In a production Semantic Cache, the costs of errors are asymmetric:

  • False Positive (Cheap): Checking the cache unnecessarily costs microseconds and fractions of a cent.
  • False Negative (Expensive): Skipping the cache and doing a full LLM inference costs seconds and actual dollars.

Therefore, we optimize to minimize False Negatives. The 8 tables / 4 planes configuration is highly viable because it guarantees we almost never miss a cache hit, while still filtering out ~39% of unnecessary lookups.

Update Cost: Adding a new vector to this LSH filter is extremely cheap ($O(1)$ local memory operations), avoiding the heavy compute required to update HNSW or IVF indexes.

Future Work (Eviction): Bloom filters don't support deletion. To handle cache eviction in production, we propose exploring **Cuckoo Filters** as a replacement to support deletion natively.
